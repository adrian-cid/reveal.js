Research What Exists

How and where to look to find modules for your functionality needs

Isolate Keywords
  • Read documentation, specs, etc...
  • Pay attention to unique nouns and verbs.
  • Ask questions!
do not make assumptions, no stupid questions

Perform Searches
  • 30k+ Modules available.
  • Favorite search engine "Drupal [keyword]".
  • Search on Drupal.org
Over 30k modules. something already exists.

Utilize the Community
  • Use IRC: #drupal, #drupal-support.
  • Use Stack Exchange: drupal.stackexchange.com.
  • Social Networks: G+, Facebook, LinkedIn.
  • Even reddit! reddit.com/r/drupal.
  • Isolate Keywords.
  • Perform Searches.
  • Utilize Community.
Community is vast, diverse & helpful.
Follow these steps 99% find module. 1% custom.

Evaluate the Options

About using project page metrics.

Read the Description
  • What does the module do?
  • What does the module not do?
  • What does the module depend on?
provides: high level overview & general understanding.
Should answer 3 questions.

Community Adoption
  • Downloads vs. Installs vs. Age.
  • Activity of issue queue.
  • Issues are good, not bad!
  • get a sense if being used.
  • Old + Low = something better.
  • Old/new + high = community using.

Maintainer Activity
  • Participate in issue queue?
  • Accepting of feedback?
  • Regular commits and/or releases?
  • Read the description.
  • Check community adoption.
  • Check maintainer activity.
If you eleminate all options, build a module.

Analyze the Gap

seeing what the module does vs. what you need it to do.

Download and Test the Module
  • Gain a better understanding.
  • Use a sandbox: simplytest.me.
  • What is offered out of the box?

Discover Missing Functionality
  • What does it not do?
  • Missing 80%? Missing 20%?
  • Bigger gap = Bigger effort.
The bigger the gap the more effort.

Check for Community Solutions
  • Look at the issue queue.
  • Search "Drupal [module] [functionality]".
  • Ask questions!
IRC, stack, etc... see if anyone has same issue.
  • Download and test the module.
  • Discover missing functionality.
  • Check for comunity solutions.
If there is no gap, use the module as is.

Determine Change

What needs to be done to close the gaps

Review the Module Code
  • Does it follow coding standards?
  • Is it extendable?
  • Can you figure out what does what?

Estimate Effort
  • How much code to rewrite?
  • How much code to add?
  • How much time is there?
Do you have enough time to complete the changes you expect?

Extend or Alter
  • Adding functionality?
  • Changing functionality?
  • Changing the modules goal?
  • Review module code.
  • Estimate effort.
  • Extend or alter?

What is a Patch?

  • drupal.org/patch.
  • A structured list of changes to files.
  • Re-appliable to files.
  • Focused on a single change.

Creating a Patch

Clone and branch module git repo.

Make changes to code.

Test your changes.

Generate patch file.

Writing a Module

  • drupal.org/developing/modules.
  • Follow Drupal coding standards.
  • Make use of hooks and APIs.
  • Test your code!

Contributing a Module

Ask yourself:

Could others use this?
Seriously, is it abstracted enough?

Name module appropriately.

  • Namespace based on dependencies.
  • Be clear, not clever.
  • Should indicate what module does.

Provide a detailed description.

  • Explain what module does and does not do.
  • Outline any dependencies.
  • Help fellow R.E.A.D-ers.

Be a good maintainer.

  • Bugs are badges not bruises.
  • Participate in the issue queue.
  • Update, release and improve.
